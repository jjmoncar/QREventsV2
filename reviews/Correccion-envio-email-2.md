Documento de Actualización de Requerimientos (ERS): GuestlyEstado: Depuración Crítica del Servicio de Correo y Autenticación de Remitente (V2.7)Proyecto: Guestly (Actualización de sistema existente)Arquitectura: Antigravity Server / SMTP Dinámico1. Resumen de la ModificaciónSe requiere una intervención técnica inmediata en el módulo de notificaciones para resolver el fallo de entrega de invitaciones. El sistema debe garantizar que el correo electrónico enviado al invitado utilice como dirección remitente el email con el cual el organizador se registró en la aplicación, asegurando la trazabilidad y la recepción efectiva.2. Corrección de Lógica: Remitente Dinámico y Validación RealActualmente, existe un fallo donde la aplicación confirma el envío ("Enviado con éxito") sin que el mensaje llegue al destinatario.Descripción del Problema: El sistema no está entregando los correos electrónicos, posiblemente debido a bloqueos de seguridad o configuraciones SMTP incorrectas. Además, no se está utilizando la identidad del organizador para el envío.Requerimiento de Corrección:Identidad del Remitente: El backend de Antigravity debe configurar dinámicamente el campo From: (De:) del correo utilizando la dirección de correo electrónico del usuario (organizador) que está enviando la invitación.Sincronización de Estado: El frontend solo debe mostrar el mensaje de éxito si el servidor SMTP confirma la aceptación del mensaje. Si el servidor de correo rechaza la petición, la app debe informar: "Error: El correo no pudo ser entregado".Eliminación de Falsos Positivos: Se debe eliminar la lógica que muestra "éxito" basándose únicamente en el clic del botón; la respuesta debe depender del código de estado real del servidor de correos.3. Especificaciones Técnicas (Antigravity Server)Configuración SMTP por Usuario: El servidor debe ser capaz de autenticar o, al menos, enmascarar el envío bajo el correo del usuario registrado.  Gestión de Entregabilidad: * El cuerpo del correo debe ser construido de forma que los proveedores (Gmail, Outlook) no lo marquen como fraude al detectar un remitente dinámico.Se debe asegurar que el código QR esté correctamente incrustado para que no sea filtrado como contenido malicioso.Logs de Error: Implementar registros en el servidor que capturen por qué un correo es rechazado (ej. "Relay access denied" o "Sender address rejected").4. Verificación de ÉxitoSe considerará corregido cuando:Al enviar una invitación, el invitado reciba el correo y vea como remitente la dirección de correo del organizador.El mensaje llegue a la bandeja de entrada (no a Spam).Si el envío falla por cualquier motivo técnico, el usuario vea un mensaje de error real en la aplicación en lugar de una confirmación falsa.
